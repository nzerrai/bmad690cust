Critiques — bloquent l'implémentation ou permettent une divergence réelle

Critique C-01 · Rubric + Adversarial S-01 S-02
Frontière generation-engine — 3 sous-questions sans réponse

Le document liste generation-engine comme bounded context distinct de dataset-api mais les place dans le même Spring Boot. Trois décisions laissées ouvertes créent des interfaces incompatibles entre builders :

Qui déclenche le job ?
Appel direct JobLauncher.run() depuis DatasetService ? Événement interne ? Endpoint séparé ? D-08 montre GET /api/datasets/{id}/status sans préciser quel package l'héberge.
PRNG non spécifié (S-01)
Builder A utilise java.util.Random(seed), Builder B ThreadLocalRandom. D-02 spécifie le triple typology_id+version+sha256 mais pas l'algorithme PRNG ni la dérivation du seed par colonne → même seed, données différentes.
Propriétaire du status (S-02)
GenerationJobListener (Batch) écrit RUNNING/COMPLETED ; DatasetService écrit PENDING/CANCELLED. Deux writers sur le même champ sans règle d'ownership → race condition sur le status.
Quota "5 jobs/user" (NFR-SC02)
Mentionné dans les cross-cutting concerns, aucun mécanisme d'enforcement défini : GenerationService ? JobExecutionListener ? Filtre ?
Corrections à apporter 1. Déclarer le déclencheur : DatasetService.startGeneration() appelle JobLauncher.run() — responsabilité dataset-api.
2. Règle PRNG : "Column seed = Long.hashCode(datasetSeed) ^ columnIndex ; PRNG = new java.util.Random(columnSeed) — aucun autre générateur autorisé."
3. Single writer : seul GenerationJobListener (Batch) écrit les transitions après soumission ; dataset-api appelle generationEngine.requestCancel(jobId) et ne touche jamais directement status.
4. Quota : GenerationService.submitJob() vérifie le compte avant création — rejette avec 429 TOO_MANY_REQUESTS si dépassé.
Critique C-02 · Adversarial S-03
DDL endpoints hors périmètre SpaceSecurityFilter — bypass RBAC
Builder sécurité
Implémente SpaceSecurityFilter couvrant /api/workspaces/{id}/** exactement selon D-05.
Builder DDL
Implémente DdlController sous /api/ddl/{did} (chemin plat naturel pour un service transversal, cohérent avec com.gendata.ddl package peer).

D-05 ne couvre pas /api/ddl/**. Les endpoints DDL bypassent le RBAC espace. Le mapping FR34–40 range DDL sous dataset-api dans un tableau et sous com.gendata.ddl dans un autre — deux builders choisissent des préfixes d'URL incompatibles.

Fix Déclarer le préfixe canonique DDL : /api/workspaces/{wid}/projects/{pid}/subprojects/{spid}/ddl/{did}/** — cela le place automatiquement sous le périmètre du SpaceSecurityFilter. Cohérent avec le routing deep-link D-12.
Critique C-03 · Adversarial S-04
Export Controller hors périmètre RBAC — fuite cross-espace possible
Builder export
Implémente ExportController à /api/export/{datasetId} avec StreamingResponseBody (D-09). Appelle DatasetRepository.findById() directement.
Builder sécurité
SpaceSecurityFilter couvre /api/workspaces/{id}/**. /api/export/** n'est pas dans le périmètre.

Un utilisateur peut exporter n'importe quel dataset par UUID sans vérification d'appartenance à l'espace. D-06 ajoute @Auditable sur export — mais c'est du logging, pas une autorisation.

Fix Imbriquer l'export sous le chemin workspace : /api/workspaces/{wid}/datasets/{did}/export — ou stipuler explicitement que ExportService doit appeler une vérification de membership identique au SpaceSecurityFilter avant de streamer.
Critique C-04 · Rubric C-02
Observabilité entièrement silencieuse

Outil bancaire avec NFR-P02 (1M lignes/min) et NFR-SC01 (1 000 users simultanés). L'observabilité est une dimension structurelle. Le document cite Spring Actuator mais ne définit rien :

  • Format de logs (JSON structuré ? Logback text ?) et niveau par défaut
  • Corrélation IDs requête HTTP → job Batch → entrée audit
  • Consommateur de /health et /metrics (Prometheus ? sonde interne ?)
  • Stack de log shipping (ELK, Loki, splunk interne ?)

Deux builders implémentant la couche logging produiront des formats incompatibles.

Fix minimal Ajouter D-17 Observabilité : format log JSON structuré (Logback + logstash-logback-encoder), MDC avec correlationId injecté sur chaque requête HTTP et propagé dans le thread Batch, Actuator exposé à l'équipe infra sur port admin séparé. Déférer la stack de shipping (Prometheus ? ELK ?) si elle n'est pas encore décidée.
Critique C-05 · Rubric C-03
Format D-01/P-01 sans champs Binds/Prevents/Rule — tooling BMad inopérant

Le document utilise le schéma D-01 / P-01 au lieu du format ARCHITECTURE-SPINE.md (AD-n avec champs structurés Binds: / Prevents: / Rule:). lint_spine.py ne peut pas valider les règles par programme, et les agents aval ne peuvent pas citer les décisions par ID stable. Critique pour le tooling BMad — pas pour la compréhension humaine du document en l'état.

Fix Soit convertir vers le format AD-n (effort moyen, gain tooling complet), soit documenter explicitement le choix de conserver le format D-01/P-01 comme variante maison et désactiver lint_spine.py pour ce projet.

Hauts — gaps significatifs, non bloquants Sprint 1

HautH-01 · Adversarial S-05
Comportement header API-Version absent non défini

D-07 dit que HandlerMapping "résout la version" depuis le header mais ne dit pas ce qui se passe si le header est absent. Builder A rejette avec 400, Builder B défault à v1. Contrat externe incompatible.

Fix Ajouter une ligne à D-07 : "Header API-Version absent → rejet 400 BAD_REQUEST, code erreur MISSING_API_VERSION."
HautH-02 · Adversarial S-06
Scope audit D-06 vs P-09 — export non couvert dans P-09

D-06 : "toutes les méthodes de génération + export" → @Auditable. P-09 règle 4 : "Toute génération de données déclenche @Auditable" — export absent. Builder implémentant ExportService depuis P-09 n'annotate pas l'export. NFR-S04 (audit 100%) non respecté.

Fix P-09 règle 4 : remplacer "génération" par "génération ou export".
HautH-03 · Rubric H-01
CORS policy non définie

WebConfig.java est mentionné comme point d'entrée CORS sans détail. Outil interne bancaire = origines strictement restreintes attendues. Sans directive, un builder crée une politique trop permissive (risque sécurité) ou trop restrictive (bloque le dev).

Fix Ajouter à D-04 ou créer D-18 CORS : origines autorisées (ex. *.banque-interne.fr prod, localhost:5173 dev uniquement), méthodes, headers autorisés.
HautH-04 · Rubric H-02
Gestion des secrets non adressée

JWT secret "configurable en env var" (D-04) — insuffisant en contexte bancaire. Aucune mention de Vault, k8s Secrets, ou solution équivalente. Builders assumera env var ; sécurité assumera autre chose.

Fix Déclarer explicitement la stratégie pour MVP (env var + recommandation Vault Phase 2) ou la solution réelle si elle existe en infra bancaire.
HautH-05 · Rubric H-03
Stratégie de migration DB (ongoing) absente

7 migrations Flyway V1–V7 listées (seed). Aucune règle pour les migrations futures : convention de nommage au-delà de V7 ? Qui peut merger une migration ? Tests de migration obligatoires ? Sans règle, deux builders produiront des migrations conflictuelles.

Fix Ajouter à P-01 ou créer P-10 Migrations : convention V{N}__{description}.sql, tests de migration obligatoires en CI, revue de migration requise pour tout changement de schéma production.

Moyens — à traiter avant Sprint 2

MoyenM-01 · Adversarial S-07
React Query — taxonomie de clés et invalidation cross-feature non définies

D-11 mandate React Query pour tout l'état serveur mais ne définit pas les formes de clés. Deux builders définissent des clés disjointes (['generation', did] vs ['datasets', wid]) — la completion d'une génération ne rafraîchit pas la liste datasets.

Fix Ajouter à D-11 un registre de clés canoniques : ['workspaces', wid], ['datasets', wid, pid, spid], ['dataset', did], ['generation', did]. Règle : on COMPLETED, invalider ['dataset', did] + ['datasets', wid, pid, spid].
MoyenM-02 · Rubric H-04
Décisions déférées dispersées sans section consolidée

WebSocket Phase 2, VIEWER Phase 2, Kubernetes Phase 3, SMTP Phase 2 sont disseminés dans le document. Un builder ne les verra pas tous et risque d'implémenter partiellement une feature Phase 2.

Fix Ajouter une section "Décisions déférées" consolidée avec condition de revisit pour chaque item.

✅  D-01 table partitionnée, D-02 versioning sha256, D-03 config immuable — trois décisions de données solides qui préviennent des classes entières de bugs.
